# Hidden Grammar / Art Lab — Architecture Reference
<!-- Written May 2026. Audience: someone who knows this project but hasn't touched it in six months. -->

---

## What This Is

The Hidden Grammar section of Artists Are Jerks is an AI-powered visual analysis tool built inside an Astro site deployed on Cloudflare Pages. Visitors upload an image, select an analytical mode and prompt, and receive a structured written analysis generated by Claude via the Anthropic API over a streaming SSE connection. The public-facing name for the tool is **Art Lab**; Hidden Grammar is the analytical framework name used in data files and internal docs only. The section lives entirely under `/hidden-grammar/` and is architecturally isolated from the rest of the AAJ site — it uses its own layout, its own data files, and its own API endpoints.

---

## Two Parallel Systems

There are two completely separate AI analysis systems in this section. They share no code and know nothing about each other.

### System 1 — Standard Art Lab
The main tool. Visitor navigates to a mode, selects a prompt, uploads an image, and receives a structured two-section analysis (IMAGE PROPERTIES + VIEWER EFFECTS). Has a follow-up interrogation window for studio chat.

- **Data source:** `src/data/analysisModes.js`
- **API endpoint:** `src/pages/api/analyze-artwork.ts`
- **UI page:** `src/pages/hidden-grammar/ai-analyze.astro`
- **Client scripts:** `src/scripts/ai-analyze/`

### System 2 — Four-Domain Analysis
A separate two-pass analysis tool at `/hidden-grammar/four-domain`. Pass 1 is pure observation; Pass 2 runs one of five audience-specific analytical formats. Has its own prompts, its own audience buttons, its own output structure. Does not use `analysisModes.js` or `analyze-artwork.ts` at all.

- **API endpoint:** `src/pages/api/four-domain.ts` — self-contained, all prompts live here
- **UI page:** `src/pages/hidden-grammar/four-domain.astro` — handles all client-side rendering and post-processing

**Critical:** Changes to `basePrompt` in `analysisModes.js` do NOT affect four-domain. Any analytical rule that needs to apply to both systems must be added to both places manually. This has caused errors. Always check both files when adding a new constraint.

## Data Layer — Three-File Coordination

Three files describe the mode inventory. They are not kept in sync automatically.

### `src/data/analysisModes.js` — Source of Truth for What's Running
The only file that controls what actually appears in the AI tool. If a mode is here, it runs. If it isn't here, it doesn't exist for the user. Contains:
- `basePrompt` — prepended to every analysis, non-negotiable, cannot be overridden by mode or prompt config
- `interrogationBase` — appended to every follow-up in the studio chat window
- `uiConfig` — controls copy button, download button, interrogation window visibility
- `analysisModes` array — all mode and prompt definitions

**When in doubt, this file wins.**

### `src/data/hg-modes.json` — Framework Reference Store
Contains the full Hidden Grammar framework content and stub/documented prompts for modes that haven't been wired into the AI tool yet. Not a production system. Useful for understanding what's planned; not authoritative about what's live.

### `MODES_REGISTRY.md` — Human-Readable Inventory
Tracks all modes across all states: LIVE, DOCUMENTED, STUB, GAP. Status definitions:
- **LIVE** — in `analysisModes.js`, functional in the tool
- **DOCUMENTED** — in `hg-modes.json` with a full prompt, not yet in the AI tool
- **STUB** — in `hg-modes.json`, prompt is a placeholder
- **GAP** — exists in source documents only, nowhere in the site

**Known desync:** As of May 2026, `analysisModes.js` contains significantly more modes than `MODES_REGISTRY.md` marks as LIVE. The registry was last updated April 2026 and does not reflect all modes added since. If you need to know what's actually running, read `analysisModes.js` directly — do not rely on the registry count.

**The gotcha:** LIVE in `MODES_REGISTRY.md` does not guarantee presence in `analysisModes.js`. STUB or GAP in the registry does not mean it's absent from `analysisModes.js`. The registry is maintained manually and can lag.

## Routing and Layout Architecture — Two Zones

The Hidden Grammar section has two zones with different layouts. Do not mix them.

### Zone A — AI Tool
All pages where visitors interact with the AI analyzer.

| Layout | `HiddenGrammarLayout` (`src/layouts/HiddenGrammarLayout.astro`) |
|---|---|
| Pages | `hidden-grammar.astro`, `hidden-grammar/for/*`, `hidden-grammar/toolkit*`, `hidden-grammar/modes/*`, `hidden-grammar/ai-analyze.astro`, `hidden-grammar/four-domain.astro` |
| Rules | Do not import AAJ fonts, color tokens, or site components. Do not use BaseLayout. |

### Zone B — Reference Pages
Framework documentation — Roots, Principles, Poles, etc.

| Layout | `BaseLayout` (normal AAJ site layout) |
|---|---|
| Pages | `hidden-grammar/framework.astro`, `hidden-grammar/roots.astro`, `hidden-grammar/principles.astro` |
| Rules | Uses AAJ design tokens and fonts normally. |

### Navigation Tiers (Zone A)
Three levels, strict order:

```
1. hidden-grammar.astro              — Art Lab landing page
2. hidden-grammar/toolkit.astro      — Tool picker by use-case (data: toolkit-modes.js)
3. hidden-grammar/toolkit/[mode].astro — Tool detail page
4. hidden-grammar/modes/[mode].astro — Prompt picker (data: analysisModes.js)
5. hidden-grammar/ai-analyze.astro   — Analyzer (mode + prompt params in URL)
```

Four-domain sits outside this flow entirely — it has its own direct URL and no entry in the toolkit navigation.

### `src/data/toolkit-modes.js`
A second data file (separate from `analysisModes.js`) that controls the Toolkit landing page cards — labels, descriptions, status badges (Live / Coming soon), and routing. A mode can be LIVE in `analysisModes.js` but have a different status in `toolkit-modes.js` if the two files aren't manually kept in sync. When adding a new mode, update both.

## The Four-Domain System in Detail

All prompts live in `src/pages/api/four-domain.ts`. There is no external data file.

### Five Audience Buttons → Five Prompt Paths

| Button | `data-value` | Prompt | Format |
|---|---|---|---|
| Artist — Work in Progress | `"artist, work in progress"` | `PASS2_PROMPT` + framing block | Full four-domain, ~8000 tokens |
| Art Historian / Educator | `"art history instructor"` | `PASS2_PROMPT` + framing block | Full four-domain, ~8000 tokens |
| Critic | `"critic"` | `CRITIC_PROMPT` | 350–450 words continuous prose |
| Docent / Public Audience | `"docent / public audience"` | `DOCENT_PROMPT` | Three-beat guide, ~400–500 words |
| Tour Guide | `"tour guide"` | `TOUR_PROMPT` | Entry prompts + three-beat, ~600–700 words |

### Pass Architecture
- **Pass 1** — Pure observation. System: "Report only what is directly present." Output streamed live to the UI, then passed to Pass 2.
- **Pass 2** — Analysis. Uses Pass 1 output as evidence base. Prompt varies by audience button.

### Domain Terminology
The four analytical domains are: PERCEPTUAL, MATERIAL AND FORMAL, CULTURAL, CONCEPTUAL. The word "register" was retired from all prompts (May 2026) — use "domain." The word "synthesis" was retired — the opening section is now "OVERVIEW."

### Post-Render JS (in `four-domain.astro`)
After `marked()` renders the model's markdown output, JavaScript normalizes headings:
- OVERVIEW / SYNTHESIS → audience-specific label (e.g. "Overview — For the Artist")
- Any `REGISTER N:` prefix stripped, heading normalized to h2
- NOISE — CONSOLIDATED normalized to h2
All via innerHTML regex replace — catches any heading level the model chooses.

## Key Decisions and Why

**`basePrompt` is non-negotiable and cannot be overridden.**
Every analysis runs through it. Mode-specific prompts operate within its constraints. If you want a rule to apply everywhere, put it in `basePrompt`. If you want it only in one mode, put it in the mode's `prompt` field. Never reference `basePrompt` in frontend files — server-side only.

**SSR via `prerender = false` on individual pages.**
The site is declared `output: 'static'` in `astro.config.mjs`, but pages and API routes that need server rendering opt in with `export const prerender = false`. The AI analyzer pages and all API endpoints use this. Pure content pages are statically rendered.

**Cloudflare adapter — `astro preview` does not work.**
The Cloudflare adapter is not compatible with `astro preview`. To test SSR behavior, you must deploy. Branch `hg-preview` auto-deploys to Cloudflare Pages — push to it and wait ~60 seconds for the deploy. This is the only reliable way to test API routes.

**The `ANTHROPIC_API_KEY` is read from `locals.runtime?.env` first, then `process.env`.**
This supports both Cloudflare's runtime environment bindings (production) and local `.env` files (development). Both API endpoints use this pattern. Do not change the order.

**Principle names are sorted by length descending before injection.**
In both `analyze-artwork.ts` and `four-domain.ts`, principle names are sorted longest-first before being injected into prompts. This ensures longer names (e.g., "Hard Edge Contrast") are matched before shorter substrings could interfere.

## Known Gotchas

**MODES_REGISTRY.md is not authoritative for what's live.** It lags. Read `analysisModes.js` to know what's actually running. See Data Layer section above.

**`basePrompt` changes don't flow into four-domain.** The two systems are completely separate. A rule added to `basePrompt` in `analysisModes.js` has no effect on `four-domain.ts`. This has caused bugs. Always ask: does this rule need to apply to both systems?

**`toolkit-modes.js` and `analysisModes.js` can desync.** A mode can be LIVE in `analysisModes.js` (and therefore functional in the analyzer) while still showing as "Coming soon" in the Toolkit, or vice versa. Both files require manual updates when a mode changes status.

**The four-domain page has no entry in MODES_REGISTRY.md.** It's a parallel system, not a mode. It won't appear in any registry count. Its architecture is documented separately in the Claude Code memory file `project_fourdomain_prompt_architecture.md`.

**Tier labels in analysis output are stripped by JavaScript, not by the prompt.** `Tier A —` and `(Tier B)` prefixes are removed post-render in `analysis.ts` via `stripTiers()`. If tier stripping stops working, look there, not at the prompts.

**The `hg-preview` branch is the deploy branch — not `main`.** Pushing to `main` does not trigger a deploy to the preview environment. Always push to `hg-preview` for Hidden Grammar changes.

## Where to Go First

**Add a new mode to the standard Art Lab:**
1. Add an entry to `analysisModes` in `src/data/analysisModes.js` — this is the only required step for the tool to work
2. Add a corresponding entry to `toolkit-modes.js` if it should appear in the Toolkit
3. Update `MODES_REGISTRY.md` status to LIVE

**Change a prompt:**
- Standard Art Lab: edit the `prompt` field in the relevant mode in `analysisModes.js`
- Four-domain: edit the relevant prompt function in `src/pages/api/four-domain.ts` — `PASS2_PROMPT`, `DOCENT_PROMPT`, `CRITIC_PROMPT`, or `TOUR_PROMPT`
- To change a rule that applies to all standard analyses: edit `basePrompt` in `analysisModes.js`
- To change a rule that applies to all four-domain analyses: edit `PASS2_PROMPT` and/or the Pass 2 system prompt in the POST handler in `four-domain.ts`

**Modify the Toolkit navigation:**
Edit `src/data/toolkit-modes.js`. The Toolkit pages (`toolkit.astro` and `toolkit/[mode].astro`) are entirely data-driven from this file.

**Add a new four-domain audience button:**
1. Add the button in `four-domain.astro` (inside `#audienceButtons`)
2. Add the `data-value` → label mapping to `overviewLabels` in the client script
3. In `four-domain.ts`: either add a new prompt function and routing condition in the POST handler, or add a new case to `getAudienceFraming()` if it uses `PASS2_PROMPT`

**Modify navigation or layout:**
Zone A pages use `HiddenGrammarLayout`. Zone B pages use `BaseLayout`. Do not mix. The layout files are at `src/layouts/`.

## Deployment

- **Deploy branch:** `hg-preview` → auto-deploys to Cloudflare Pages on push
- **Deploy command:** `git push origin hg-preview`
- **Preview server:** `npm run dev` works for static pages; use deploy for anything with `prerender = false`
- **`astro preview` does not work** with the Cloudflare adapter
- **Environment variable:** `ANTHROPIC_API_KEY` must be set in Cloudflare Pages environment settings (not just `.env`) for the API routes to work in production
